# D2C 口红实验室 - 项目介绍

## 项目概述

**D2C 口红实验室**是一个面向美妆行业的智能化趋势分析与色号设计系统。项目采用 B/S 架构，前后端分离设计，打通产品、研发、市场、运营四大核心职能，形成 D2C（Direct-to-Consumer）闭环协作模型。

系统核心解决三个问题：
1. **产品部门**：如何快速捕捉社交媒体上的美妆趋势？
2. **研发部门**：如何将趋势数据转化为可落地的色号设计？
3. **市场/运营部门**：如何验证设计并生成宣发内容？

---

## 核心功能模块

### 功能一：社交数据捕获与语义解构（产品部门）

自研小红书爬虫，完全 Python 实现，无需依赖第三方服务。

**技术亮点：**
- **签名算法逆向**：破解小红书 API 签名机制，实现自定义 Base64 编码表、CRC32 变体算法、无符号右移运算的纯 Python 实现
- **双签名策略**：Playwright 页面签名 + Python 算法备用，确保高可用性
- **数据融合**：笔记内容与评论数据关联存储，为后续分析提供完整语料

**输出**：结构化 JSON 数据（笔记标题、内容、点赞数、评论列表等）

---

### 功能二：数字化调色板与文化命名（研发部门）

这是项目的核心创新点，将 AI 分析结果与专业设计工具结合。

**技术亮点：**

#### 1. Agent DAG 执行引擎
- 8 个专业工具（load_data、analyze_tone、analyze_style、generate_chart、generate_text、critic、generate_report、generate_color_schemes）
- 拓扑排序识别依赖关系，无依赖步骤并行执行
- WebSocket 实时推送执行状态，前端可看到每一步的进度和结果

#### 2. RAG 自我修正机制
- LLM 分析时常出现占位符（如"妆容风格1"、"色调"）
- 实现自动验证：检测占位符、无效字段、数据类型
- 错误反馈注入 Prompt，最多 3 次重试自我修正

#### 3. Fabric.js 调色板（800 行自研组件）
- **颜色混合**：离屏 Canvas 重新渲染，计算交集区域像素平均色，真实模拟口红叠涂效果
- **取色器**：读取 lowerCanvas 像素，正确处理 devicePixelRatio，支持透明度叠加后的颜色读取
- **Figma 风格工具栏**：自适应缩放、图层上下移动、网格对齐

#### 4. 双模型策略
- GPT-4o-mini 处理常规任务（笔记分析、文案生成）
- GPT-4o 处理复杂结构化输出（ECharts 图表配置）
- 成本与质量的平衡

**输出**：
- 趋势报告（含色调分布、风格分析、关键词词云、数据图表）
- 配色方案（3 组带 HEX 值的专业配色）
- 色号命名（基于诗经/唐诗/宋词风格）

## 核心技术难点

### 1. 小红书签名算法逆向

小红书 API 使用自定义签名机制防止爬虫：
- **自定义 Base64 字符表**：标准 Base64 字符被完全打乱顺序
- **CRC32 变体算法**：非标准 CRC32，包含特定的异或常量
- **无符号右移运算**：Python 默认有符号右移与 JavaScript 行为不同，需用 ctypes 模拟

实现完全自主可控，不依赖第三方签名服务。

### 2. Agent 并行执行与状态同步

挑战：
- 步骤间有依赖关系（如 generate_chart 依赖 analyze_tone）
- 需要实时推送状态到前端
- 支持暂停/恢复执行

解决方案：
- DAG 拓扑排序识别可并行步骤
- asyncio.gather 并行执行
- WebSocket 广播状态更新

### 3. 颜色混合的真实效果

挑战：
- 直接取色块的 fill 颜色无法反映透明度叠加
- 需要模拟真实的口红叠涂效果

解决方案：
- 创建离屏 Canvas
- 按 z-order 顺序绘制每个色块（含透明度）
- 计算交集区域所有像素的平均色

### 4. 三层存储降级架构

挑战：
- 开发环境不想配置 Redis
- 生产环境需要高性能和高可用
- 历史数据需要持久化

解决方案：
- L1: Redis（热数据，多实例共享）
- L2: 内存字典（开发/测试降级）
- L3: PostgreSQL（冷数据持久化）

读取优先级：Redis → 内存 → PostgreSQL
写入策略：Redis/内存同步写入，PostgreSQL 异步写入
